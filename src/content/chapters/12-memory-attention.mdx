export const codeCutPoint = `// findCutPoint — 从最新消息向前回溯，保留 keepRecentTokens
export function findCutPoint(
  entries: SessionEntry[],
  startIndex: number,
  endIndex: number,
  keepRecentTokens: number,
): CutPointResult {
  const cutPoints = findValidCutPoints(entries, startIndex, endIndex);
  if (cutPoints.length === 0) {
    return { firstKeptEntryIndex: startIndex, turnStartIndex: -1, isSplitTurn: false };
  }

  let accumulatedTokens = 0;
  let cutIndex = cutPoints[0];

  // 从尾部向头部走，累加 token 估算
  for (let i = endIndex - 1; i >= startIndex; i--) {
    const entry = entries[i];
    if (entry.type !== "message") continue;
    const messageTokens = estimateTokens(entry.message);
    accumulatedTokens += messageTokens;

    // 超出预算 → 找最近的合法切割点
    if (accumulatedTokens >= keepRecentTokens) {
      for (let c = 0; c < cutPoints.length; c++) {
        if (cutPoints[c] >= i) { cutIndex = cutPoints[c]; break; }
      }
      break;
    }
  }
  // 永远不在 toolResult 处切割 — 它必须跟在 tool call 后面
  return { firstKeptEntryIndex: cutIndex, turnStartIndex, isSplitTurn };
}`;

export const codeTokenEstimate = `// estimateTokens — chars/4 启发式，覆盖所有消息类型
export function estimateTokens(message: AgentMessage): number {
  let chars = 0;
  switch (message.role) {
    case "user": {
      const content = message.content;
      if (typeof content === "string") chars = content.length;
      else if (Array.isArray(content)) {
        for (const block of content) {
          if (block.type === "text" && block.text) chars += block.text.length;
        }
      }
      return Math.ceil(chars / 4);  // 保守估算（略偏高）
    }
    case "assistant": {
      for (const block of message.content) {
        if (block.type === "text") chars += block.text.length;
        else if (block.type === "thinking") chars += block.thinking.length;
        else if (block.type === "toolCall")
          chars += block.name.length + JSON.stringify(block.arguments).length;
      }
      return Math.ceil(chars / 4);
    }
    case "toolResult": {
      // 图片特殊处理：每张 ~1200 tokens
      for (const block of message.content) {
        if (block.type === "text") chars += block.text.length;
        if (block.type === "image") chars += 4800;
      }
      return Math.ceil(chars / 4);
    }
  }
  return 0;
}`;

## 为什么会痛：状态鸿沟与注意力鸿沟

回顾 Ch9 的理论框架，这一章的痛点可以精确地映射到两条鸿沟。

**状态鸿沟：LLM 是无状态函数。** 每次调用都是一次独立的 `f(context) → response`。上一次调用的"记忆"完全依赖你在 context 中放了什么。关闭终端 = 丢失 context = 丢失所有状态。没有任何东西是"自动记住"的——记忆完全是 Harness 的工程责任。

**注意力鸿沟：窗口有限，注意力有偏。** 即使上下文没满，LLM 的注意力分布也不均匀。研究表明存在 "lost in the middle" 现象——窗口头尾的信息被重点关注，中间的信息容易被忽略。当 Context Window 接近满载时，这个问题更加严重：质量在窗口填满之前就已经开始退化了。

这两条鸿沟引出了本章的**中心轴线**——成本反转原理在记忆领域的具体展开：

**在 LLM 上下文中，保留是昂贵的，取回是廉价的。** 每一个常驻上下文的 token 都在消耗注意力预算、推高推理成本、挤压新信息的空间。而一次工具调用（`read`、`grep`、`glob`）只需要毫秒级延迟和极低的 token 开销。传统计算机科学里"cache everything"的直觉，在 LLM 世界中是反模式。

## 记忆层次结构

就像 CPU 有 L1/L2/L3 缓存和主存，Agent 的记忆也有层次。每一层的延迟、容量和持久性不同，对应不同的工程策略。

| 层级 | 名称 | 载体 | 容量 | 持久性 | 延迟 | Pi 实现 |
|------|------|------|------|--------|------|---------|
| **L1** | 工作记忆 | Context Window 中的最近消息 | ~20K tokens | 单 turn | 0 | `keepRecentTokens: 20000` |
| **L2** | 压缩记忆 | Compaction Summary | ~800 tokens | 跨 turn（同会话） | 0 | `CompactionEntry.summary` |
| **L3** | 持久记忆 | Session 文件（JSONL 树） | 无限 | 跨重启 | 1 次 read | `SessionManager` |
| **L4** | 外部记忆 | 文件系统、git、CLAUDE.md | 无限 | 跨会话 | 1 次工具调用 | `read`/`glob`/`bash` 工具 |

关键数据：Pichay 等人的研究 (2025) 在大规模 Agent 工作流中测量到 **21.8% 的上下文空间被浪费的历史消息占据**，而 token 放大率（实际消耗 / 有效信息量）高达 **84.4x**。这意味着 Agent 每处理 1 token 的有效信息，就要为 84 token 的冗余上下文买单。

这组数据证明了记忆层次结构不是优化——是生存需求。没有压缩和分层，Agent 在复杂任务中根本无法高效运作。

## 成本反转全景：保留 vs 取回的每一个决策

Ch9 提出了成本反转原理的总体框架。在记忆与注意力领域，这个原理变得具体而可操作——Harness 设计者在每一个子系统中都需要在"保留"和"取回"之间做出选择。

下表展示了 Pi 在每个维度上的具体决策：

| 维度 | "保留"策略（昂贵） | "取回"策略（廉价） | Pi 的选择 |
|------|-------------------|-------------------|-----------|
| **历史消息** | 全量保留在 context 中 | Compaction 摘要 + keepRecentTokens | 取回：只保留最近 20K tokens，其余压缩为 ~800 token 摘要 |
| **工具结果** | 大段输出原样保留 | 截断 + 按需重读 | 取回：`truncateHead` 50KB / `truncateTail` 100KB，超出部分丢弃 |
| **Thinking** | 推理过程全量保留 | 仅保留结论 | 取回：thinking block 不参与后续上下文（API 层面已过滤） |
| **文件内容** | 预加载所有相关文件 | `read` 工具按需读取 | 取回：从不预加载，Agent 自行决定何时读取 |
| **子任务上下文** | 全部在主 Agent context 中 | 子 Agent 隔离执行，只返回摘要 | 取回：subagent 模式隔离上下文，仅回传结果 |

**每一行的默认选择都是"取回"。** 这不是偶然——这是成本反转原理的直接推论。在 LLM 上下文中，空间 = 注意力 = 质量。节省空间就是提升质量。

## 三种压缩策略

当上下文不可避免地增长时，如何压缩？业界收敛到三种策略，各有优劣。

| 策略 | 原理 | 信息保真度 | 实现复杂度 | 代表 |
|------|------|-----------|-----------|------|
| **Token 清除** | 直接截断/删除旧消息 | 低（信息丢失） | 低 | 简单 FIFO 队列 |
| **LLM 摘要** | 用 LLM 总结被丢弃的消息 | 中（结构化保留关键信息） | 中 | Pi Compaction、Claude Code |
| **Observation 遮蔽** | 选择性隐藏工具输出，保留结构 | 高（对话骨架完整） | 高 | Factory.ai |

Pi 选择了 **LLM 摘要**策略，并添加了三个关键增强：

1. **结构化模板**：不是自由摘要，而是强制 Goal / Constraints / Progress / Decisions / Next Steps 格式（见上方 `codeCompactionPrompt`），确保关键信息不被遗漏。

2. **增量更新**：第二次及以后的压缩不从零开始——它接收上一次的摘要作为 `<previous-summary>`，用 `UPDATE_SUMMARIZATION_PROMPT` 合并新信息。这避免了反复压缩导致的信息衰减。

3. **文件操作追踪**：每次压缩自动提取并附加 `readFiles` / `modifiedFiles` 列表，让 Agent 在压缩后仍然知道哪些文件被读过、哪些被改过——这是最容易在压缩中丢失的信息。

## 会话持久化与分支

解决"关闭终端 = 丢失一切"的方法听起来简单——把对话存下来。但一旦你支持"分支"（用户从历史中某个点重新出发），简单的数组就不够用了。

**Pi 的方案：把会话存为一棵树。** 这不是比喻——它的数据模型就是 git 的简化版。

**2. Append-only 持久化。** Session 文件是 JSONL 格式——每一行是一个 JSON 对象，新 Entry 直接 append 到文件末尾。没有 update，没有 delete。这意味着即使进程崩溃，已写入的数据也不会损坏。

**3. 上下文重建 = 从叶到根的路径遍历。** `buildSessionContext()` 从当前叶节点向根节点回溯，收集路径上的所有 Entry，就像 `git log` 从 HEAD 沿 parent 指针回溯。分支、压缩、模型切换——所有历史事件都在路径上以 Entry 的形式存在。

ContextBranch (Becker et al., 2025) 的研究验证了这种树形结构的价值：在复杂推理任务中，支持分支的 Agent 比线性历史的 Agent 在 Reasoning Depth 上高出 **40%**——因为分支让 Agent 能够探索多条路径而不丢失之前的上下文。

## Pi 的解法：会话管理深度解剖

前面我们建立了理论框架——状态鸿沟、注意力鸿沟、成本反转、记忆层次。现在让我们钻进 Pi 的源码，看看这些理论如何落地为工程实现。Pi 的会话管理由三个核心模块驱动：SessionManager（1,410 行）负责树形存储与分支；Compaction（815 行）负责上下文压缩；Branch Summarization（355 行）负责分支切换时的上下文保留。让我们逐一解剖关键设计决策。

### JSONL 树状存储

Pi 把整个会话存为一个 JSONL 文件——每行一个 JSON 对象，每个对象是一个 `SessionEntry`。前面我们已经看过 9 种 Entry 类型，它们统一继承 `SessionEntryBase`，通过 `id` / `parentId` 构成一棵树。

**为什么是 append-only JSONL？** 这是状态鸿沟的最安全桥接方式。崩溃不会丢失已写入的数据——最坏情况是最后一行不完整，加载时跳过即可。没有 update，没有 delete，没有 WAL，没有事务。简单到不可能出错。

Pi 还做了一个巧妙的优化：**lazy write**。新会话不会立即写入文件，而是缓存在内存中，直到收到第一条 assistant 响应才真正 flush 到磁盘。这避免了用户输入一句话就关闭窗口时创建无用的 session 文件。

### 分支机制

分支的全部实现就是移动一个指针。没有数据复制，没有删除操作。旧分支的所有 Entry 原封不动地保留在 JSONL 文件中。

当用户切换到另一个分支时，`branchWithSummary()` 会为被离开的路径生成一条摘要 Entry。这样新分支的上下文中包含了旧路径的概要——不需要重读整个旧分支，一条摘要就够了。

### Compaction 系统

Compaction 是 Pi 记忆系统的核心——它决定了"丢弃什么、保留什么、如何摘要"。整个流程分为两个阶段，可以并行执行：**历史消息压缩**（生成摘要替代旧消息）和 **Turn 前缀压缩**（处理当前 turn 中已完成的工具调用）。

**切点选择**是 Compaction 中最关键的细节。`findCutPoint` 从最新消息向前回溯，累加 token 估算，直到超出 `keepRecentTokens` 预算，然后找到最近的合法切割点。

**为什么不能在 toolResult 处切割？** 因为 `toolResult` 和 `toolCall` 必须配对——单独的 `toolResult` 语义不完整。LLM 看到一个没有对应 `toolCall` 的 `toolResult` 会困惑，轻则输出质量下降，重则产生幻觉。`findValidCutPoints` 会排除所有 `toolResult` 位置，确保切割点总是落在完整的 turn 边界上。

`chars / 4` 是成本反转的直接推论——这里的不对称性是：高估 token 数（低成本，只是提前触发一次 compaction）vs 低估 token 数（高成本，上下文溢出导致 API 报错或质量崩塌）。用一个偏高的常数因子，把风险推向低成本的一侧。

结构化格式不是为了美观——它是强制保留的机制。自由格式的摘要容易遗漏"已完成的任务列表"或"关键决策的理由"，而模板中的每个 section 都是一个不可跳过的 slot。

文件操作追踪是另一个关键增强。每次 Compaction 后，摘要会附带两个 XML 标签：`<read-files>` 列出被读过的文件，`<modified-files>` 列出被修改过的文件。这解决了压缩后最常见的问题——Agent 忘记自己读过/改过哪些文件，开始重复读取。

### 何时触发

Pi 使用双阈值策略：当上下文占用超过窗口的 **75%** 时触发 compaction，而不是等到 95%。为什么提前这么多？

因为注意力鸿沟——LLM 的性能在窗口填满之前就已经开始退化了。"Lost in the middle" 效应意味着，当窗口占用 80% 时，中间的信息已经很难被有效利用。提前在 75% 触发 compaction，反而延长了生产性会话长度。如果等到 95%，Agent 可能已经在 85% 时就开始重复工作了。

更极端的情况下（比如单条消息就很大），如果 compaction 后仍然超过 **95%**，Pi 会触发更激进的二次压缩。

### 会话迁移

Pi 的 session 文件格式经历了三个版本：v1（基础消息列表）→ v2（添加树结构）→ v3（添加扩展 Entry 类型）。每次版本升级都保持向后兼容——旧格式文件在加载时自动运行迁移函数，升级到最新格式后继续正常使用。用户无感知，不丢数据。

## 保留什么、丢弃什么

Compaction 的本质是一个决策问题：哪些信息值得占据宝贵的上下文空间？经过 Pi 的实践和社区的经验积累，答案逐渐收敛。

**永远保留的信息：**
- **系统指令和用户偏好** — 丢失它们意味着 Agent 行为漂移
- **未解决的问题和当前目标** — 丢失它们意味着 Agent 迷失方向
- **文件路径和操作记录** — 丢失它们意味着重复劳动（`readFiles` / `modifiedFiles`）
- **错误信息和关键决策** — 丢失它们意味着重蹈覆辙

**优先丢弃的信息：**
- **旧的工具输出**（token 放大率 84.4x）— 一次 `read` 调用就能取回
- **中间推理过程** — 只保留结论，过程可以重新推导
- **重复的文件内容** — 同一个文件被读了三遍，只需保留最新版本的摘要

**外部记忆的补充：** 当信息太重要不能丢、但又太大不能留在上下文中时，写入外部存储。Pi 生态中常见的模式包括：Initializer-Executor pattern（初始化 Agent 将规划写入 `todo.md`，执行 Agent 按 todo 列表逐项完成）、git commit messages（每次提交都是一条持久化的进度记录）、以及 `CLAUDE.md`（跨会话的项目级记忆）。

## 不同产品为什么做了不同选择

记忆系统没有唯一正确的设计——不同产品面对不同的约束，做出了不同的权衡。

| 维度 | Pi | Claude Code | LangGraph | Manus |
|------|-----|-------------|-----------|-------|
| **存储格式** | JSONL 树（append-only） | 内存 + 文件系统 | 检查点图（StateGraph） | 沙盒文件系统 |
| **分支** | 原生支持（tree） | 无 | 原生支持（StateSnapshot） | 无 |
| **压缩策略** | LLM 摘要 + 结构化模板 | LLM 摘要 | 用户自定义 reducer | Observation 遮蔽 |
| **触发时机** | 75% / 95% 双阈值 | 自适应 | 用户控制 | 按步骤 |
| **迁移** | v1→v2→v3 自动迁移 | N/A | Schema 版本控制 | N/A |
| **跨会话** | Session 文件 + CLAUDE.md | CLAUDE.md | 外部数据库 | 任务日志 |

关键观察：**越是交互式的产品（Pi、Claude Code），越倾向于自动化的压缩和透明的持久化。** 越是框架型的产品（LangGraph），越倾向于把控制权交给开发者。没有对错——只有场景适配。

## 元技能：如何判断"这是记忆问题"

**症状识别：** Agent 重复读取已处理过的文件、遗忘之前的决策、在长会话后期输出质量明显下降、重启后无法恢复进度——这些都是记忆问题的典型信号。

**优先级排序：** 先实现 compaction，再考虑分支。Compaction 解决的是"Agent 能不能跑完长任务"，分支解决的是"Agent 能不能高效探索"。前者是生存需求，后者是效率优化。

**评估方法：** 让 Agent 执行一个需要 30+ 轮对话的任务（比如迁移 10 个文件的 API 调用），观察它在第 20 轮时是否还记得第 5 轮的决策。如果不记得，你需要 compaction。如果 compaction 后摘要丢失了关键信息，你需要结构化模板。

---

## 简化思考

更大的上下文窗口（200K → 1M → 10M）会降低 compaction 的触发频率，但不会消除对它的需求。原因有二：第一，token 成本与窗口大小线性相关——10M 窗口意味着每次调用的成本是 200K 窗口的 50 倍；第二，context rot 不等于 context overflow。即使窗口没满，"lost in the middle" 效应仍然存在。更大的窗口只是把问题从"放不下"变成了"找不到"。

Compaction 不仅仅是空间管理——它是注意力管理。

## 检验标准

- [ ] 关闭重启后从 JSONL 恢复完整对话
- [ ] fork() 后新分支消息不影响原分支
- [ ] getBranch() 返回 root 到 leaf 的正确路径
- [ ] 上下文超阈值时自动触发 compaction
- [ ] Compaction 后包含摘要 + 保留的最近消息
- [ ] 切点不在 toolResult 处
- [ ] 摘要中包含 read-files 和 modified-files 列表

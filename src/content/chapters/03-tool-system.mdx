**content vs details 的分离是核心设计决策**：

- `content` 是发送给 LLM 的内容，直接影响 LLM 的下一步决策
- `details` 是 UI 元数据，用于渲染、日志、调试，但 **不会发送给 LLM**

这意味着工具可以告诉 LLM "Successfully replaced text in file.ts"（简洁的 content），同时给 UI 发送完整的 diff 预览（丰富的 details）。LLM 不需要处理大量的 diff 信息，UI 却能展示完整的变更。

### 为什么需要 Schema 验证？

因为 LLM 的输出不可信。

你定义了一个 bash 工具，参数是 `{command: string}`。但 LLM 可能返回 `{command: 42}` 或 `{cmd: "ls"}` 或空对象 `{}`。没有验证，你的 executeTool 会在运行时崩溃。

Pi 用 TypeBox（不是 Zod）来定义 schema。为什么？因为 TypeBox 直接生成标准 JSON Schema，LLM 需要这个格式来理解工具参数。Zod 需要额外转换才能生成 JSON Schema。

### 为什么选择 TypeBox 而不是 Zod？

Pi 使用 `@sinclair/typebox` 而不是更流行的 Zod 来定义工具参数 Schema。原因很直接：

```typescript
import { Type, type Static } from '@sinclair/typebox';

// TypeBox：定义一次，自动得到 JSON Schema + TypeScript 类型
const bashSchema = Type.Object({
  command: Type.String({ description: "Bash command to execute" }),
  timeout: Type.Optional(Type.Number({ description: "Timeout in seconds" })),
});

// Static<T> 从 Schema 推导出 TypeScript 类型
type BashInput = Static<typeof bashSchema>;
// 等价于: { command: string; timeout?: number }
```

**TypeBox 直接生成标准 JSON Schema**，这正是 LLM API（OpenAI、Anthropic 等）需要的格式。Zod 虽然 API 更优雅，但需要额外的 `zod-to-json-schema` 转换步骤，且转换可能丢失 `description` 等元数据。

对于 Agent 工具系统来说，JSON Schema 不是可选的附加功能，而是与 LLM 通信的核心协议。TypeBox 的 "Schema-first" 设计正好匹配这个需求。

### AJV 验证：类型安全的最后防线

LLM 返回的参数是 JSON，可能存在类型错误。Pi 使用 AJV 进行运行时验证：

<SourceReader
  client:visible
  code={`// Create a singleton AJV instance with formats
const ajv = new Ajv({
  allErrors: true,   // 收集所有错误，而非在第一个错误时停止
  strict: false,     // 允许 TypeBox 的扩展关键字（如 description）
  coerceTypes: true,  // "42" → 42 自动类型转换
});
addFormats(ajv);

export function validateToolArguments(tool: Tool, toolCall: ToolCall): any {
  // Compile the schema.
  const validate = ajv.compile(tool.parameters);

  // Clone arguments so AJV can safely mutate for type coercion
  const args = structuredClone(toolCall.arguments);

  // Validate the arguments (AJV mutates args in-place for type coercion)
  if (validate(args)) {
    return args;
  }

  // Format validation errors nicely
  const errors = validate.errors
    ?.map((err: any) => {
      const path = err.instancePath
        ? err.instancePath.substring(1)
        : err.params.missingProperty || "root";
      return \`  - \${path}: \${err.message}\`;
    })
    .join("\\n") || "Unknown validation error";

  throw new Error(
    \`Validation failed for tool "\${toolCall.name}":\\n\${errors}\`
  );

三个关键配置的设计考量：

1. **`coerceTypes: true`** — LLM 有时会把数字作为字符串返回（`"42"` 而不是 `42`），开启类型强制转换后 AJV 自动处理这种差异
2. **`allErrors: true`** — 如果参数有 3 个错误，一次性全部报告，让 LLM 在下次重试时能修正所有问题
3. **`strict: false`** — TypeBox 在 JSON Schema 中添加了 `description`、`default` 等扩展字段，strict 模式会拒绝这些字段

## 3.2 内置工具解剖

Pi 内置了 7 个工具，覆盖了 coding agent 的核心操作：

| 工具 | 功能 | 关键特性 |
|------|------|----------|
| `bash` | 执行 shell 命令 | 进程树清理、输出截断、超时 |
| `edit` | 精确文本替换 | 模糊匹配、唯一性检查、BOM 保留 |
| `write` | 创建/覆盖文件 | 文件变更队列 |
| `read` | 读取文件内容 | 头部截断 |
| `grep` | 搜索文件内容 | 行长截断 |

| `ls` | 列出目录 | 简洁输出 |

接下来深入剖析 bash 和 edit —— 两个最复杂也最核心的工具。

**进程树清理**：bash 命令可能启动子进程（如 `npm start` 启动 node 服务），简单地 kill 主进程会留下僵尸子进程。Pi 通过 `killProcessTree` 杀掉整个进程树：

```typescript
// spawn 使用 detached: true 创建新进程组
const child = spawn(shell, [...args, command], {
  cwd,
  detached: true,    // 创建新进程组，便于整体清理
  env: env ?? getShellEnv(),
  stdio: ["ignore", "pipe", "pipe"],
});

// 超时或取消时，杀掉整个进程树
const onAbort = () => {
  if (child.pid) {
    killProcessTree(child.pid);  // 不是 child.kill()，而是整个进程树
  }
};
```

**输出截断策略**：LLM 有上下文窗口限制，不能把 10MB 的构建日志全部塞进去。Pi 采用**滚动缓冲区 + 尾部截断**策略：

<SourceReader

export const DEFAULT_MAX_LINES = 2000;
export const DEFAULT_MAX_BYTES = 50 * 1024; // 50KB

// Bash 工具的输出处理
const chunks: Buffer[] = [];
let chunksBytes = 0;
const maxChunksBytes = DEFAULT_MAX_BYTES * 2;  // 保留 2 倍，确保截断有足够素材

const handleData = (data: Buffer) => {
  totalBytes += data.length;

  // 超过 50KB 时开始写入临时文件
  if (totalBytes > DEFAULT_MAX_BYTES && !tempFilePath) {
    tempFilePath = getTempFilePath();
    tempFileStream = createWriteStream(tempFilePath);
    for (const chunk of chunks) {
      tempFileStream.write(chunk);
    }
  }

  // 写入临时文件
  if (tempFileStream) {
    tempFileStream.write(data);
  }

  // 维护滚动缓冲区
  chunks.push(data);
  chunksBytes += data.length;

  // 淘汰旧的 chunk
  while (chunksBytes > maxChunksBytes && chunks.length > 1) {
    const removed = chunks.shift()!;
    chunksBytes -= removed.length;
  }
};`}

      index: exactIndex,

      usedFuzzyMatch: false,

  const fuzzyContent = normalizeForFuzzyMatch(content);
  const fuzzyOldText = normalizeForFuzzyMatch(oldText);

    index: fuzzyIndex,

    usedFuzzyMatch: true,

### 可插拔操作模式（Pluggable Operations Pattern）

注意 Pi 的 bash 工具不直接调用 `child_process.exec`，而是通过 `BashOperations` 接口间接调用。这种**可插拔操作模式（Pluggable Operations Pattern）**让同一个工具定义能在不同环境下执行：本地直接运行、SSH 远程执行、Docker 沙箱执行。你只需要替换 Operations 实现，不用改工具逻辑。

bash 和 edit 工具都定义了 Operations 接口（`BashOperations`、`EditOperations`）：

通过替换 Operations 实现，同一个工具可以运行在完全不同的环境：

- **本地执行**：默认实现，直接调用 Node.js 的 `child_process` 和 `fs`
- **SSH 远程执行**：Operations 通过 SSH 连接转发命令和文件操作
- **Docker 沙箱**：Operations 在容器内执行，隔离风险操作
- **测试 Mock**：Operations 返回预设结果，无需真实文件系统

### 文件变更队列：同文件串行，不同文件并行

当 LLM 一次请求多个 edit 操作时，如果两个 edit 修改同一个文件，并行执行会导致竞态条件。Pi 通过 `withFileMutationQueue` 解决这个问题：

<SourceReader
  client:visible
  code={`const fileMutationQueues = new Map<string, Promise<void>>();

async function getMutationQueueKey(filePath: string): Promise<string> {
  const resolvedPath = resolve(filePath);
  try {
    return await realpath(resolvedPath);  // 解析符号链接
  } catch {
    return resolvedPath;
  }
}

export async function withFileMutationQueue<T>(
  filePath: string, fn: () => Promise<T>
): Promise<T> {
  const key = await getMutationQueueKey(filePath);
  const currentQueue = fileMutationQueues.get(key) ?? Promise.resolve();

  let releaseNext!: () => void;
  const nextQueue = new Promise<void>((resolveQueue) => {
    releaseNext = resolveQueue;
  });
  const chainedQueue = currentQueue.then(() => nextQueue);
  fileMutationQueues.set(key, chainedQueue);

  await currentQueue;
  try {
    return await fn();
  } finally {
    releaseNext();
    if (fileMutationQueues.get(key) === chainedQueue) {
      fileMutationQueues.delete(key);
    }
  }
}`}

## 3.3 工具调用完整链路

当 LLM 返回包含工具调用的消息后，Pi 的 agent-loop 开始执行一个精心编排的三阶段流水线。

### 三阶段流水线

  <VideoEmbed
  client:visible
  title="V08 · Tool Call Flow"
  description="工具调用的三阶段流水线动画 — 准备、执行、完成"
  duration="~18s"
  compositionId="ToolCallFlow"
/>

整个流程分为三个阶段：

**Phase 1：Prepare（顺序执行）**

```
对每个 toolCall 依次执行：
  1. 查找工具 → 找不到则立即返回错误
  2. validateToolArguments() → 验证参数
  3. beforeToolCall 钩子 → 可以 block 执行
  → 输出：PreparedToolCall（可执行） 或 ImmediateToolCallOutcome（立即结果）
```

**Phase 2：Execute（并发执行）**

```
所有 PreparedToolCall 同时启动执行：
  1. tool.execute() 运行
  2. onUpdate 回调流式发送进度
  3. await Promise.all(updateEvents) 确保进度事件全部发出
  → 输出：ExecutedToolCallOutcome（成功或错误）
```

**Phase 3：Finalize（源序处理）**

```
按 LLM 返回的原始顺序依次处理：
  1. afterToolCall 钩子 → 可以修改结果
  2. emitToolCallOutcome → 发出 tool_execution_end 事件
  3. 构建 ToolResultMessage → 发出 message_start/end
```

### 核心代码：并行执行保序

## 3.4 工具执行的边界情况

理解正常流程后，真正的技术深度在边界情况的处理上。

### Update Then Fail

```
工具执行过程：
  onUpdate({ text: "编译中..." })       ← UI 收到了
  onUpdate({ text: "编译 50% 完成" })    ← UI 收到了
  onUpdate({ text: "编译 80% 完成" })    ← UI 收到了
  throw new Error("编译失败: 语法错误")   ← 工具抛错

结果：
  3 个 tool_execution_update 事件已经发送（不可撤回）
  1 个 tool_execution_end 事件，isError: true
  LLM 收到的是错误消息，不是之前的进度更新
```

这不是 bug，而是有意的设计。进度事件是给 UI 的实时反馈，工具结果是给 LLM 的最终判断依据。两者独立。

### beforeToolCall 的时序

```
LLM 请求调用工具
  → validateToolArguments()    ← 参数验证（先执行）
  → beforeToolCall()           ← 钩子（后执行，收到已验证的参数）
  → tool.execute()             ← 执行
```

**beforeToolCall 不能修改参数**。它收到的 `args` 是只读的（虽然 TypeScript 接口没有用 `readonly` 标记，但修改它不会影响后续传给 `execute` 的参数，因为它们来自同一个 `validatedArgs` 引用）。

beforeToolCall 唯一能做的事情是：**决定是否阻止执行**。返回 `{ block: true, reason: "..." }` 会阻止工具执行并将 reason 作为错误消息发送给 LLM。

### ToolResult 事件与 Assistant 事件的区别

```
AssistantMessage 事件序列：
  message_start → message_update → message_update → ... → message_end
                   （流式更新，可能有很多次）

ToolResultMessage 事件序列：
  message_start → message_end
  （没有 update 阶段，同步完成）
```

工具执行的进度更新通过 `tool_execution_update` 事件发送，而不是 `message_update`。这是因为 `ToolResultMessage` 是在工具执行完成后一次性构建的。

## 3.6 本章小结

### 核心要点

1. **工具系统 = Schema 定义 + 验证 + 执行 + 钩子** —— 四个环节缺一不可，构成了 Agent 与外部世界交互的完整管道

2. **并行执行保序是关键设计** —— `.map()` 启动所有 Promise，`for...of await` 按源序收集结果。这个模式在 Agent 系统中至关重要：LLM 必须按请求顺序看到结果

3. **可插拔 Operations 接口使工具与底层解耦** —— 同一个 bash/edit 工具可以运行在本地、SSH、Docker、测试 Mock 等不同环境

4. **content vs details 分离** —— 发送给 LLM 的信息（content）和展示给用户的信息（details）有不同的需求，强制分离避免信息过载

5. **TypeBox + AJV 的选择** —— 不是"比 Zod 更好"，而是"直接生成 LLM 需要的 JSON Schema"，减少转换步骤和信息损失

### 架构关系图

```
AgentLoopConfig
  ├── tools: AgentTool[]           ← 注册的工具集合
  ├── toolExecution: "parallel"    ← 执行模式
  ├── beforeToolCall()             ← 准备阶段钩子
  └── afterToolCall()              ← 完成阶段钩子

工具调用流水线:
  LLM 返回 toolCall[]
    → [Phase 1] prepare: find → validate → beforeToolCall
    → [Phase 2] execute: tool.execute() (并发)
    → [Phase 3] finalize: afterToolCall → emit (按序)
    → ToolResultMessage[] 送回 LLM
```

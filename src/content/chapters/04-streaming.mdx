## 4.1 EventStream 核心

### 问题：异步生产者与消费者如何协调？

LLM 的流式响应是一个典型的 "生产者-消费者" 问题：LLM 以不可控的速率产出 token，UI 需要按自己的节奏消费。同时，使用者可能需要两种消费模式：逐事件处理（实时 UI 更新）或等待最终结果（获取完整消息）。

Pi 通过 `EventStream<T, R>` 泛型类解决这个问题。`T` 是事件类型，`R` 是最终结果类型。

### 源码解析

<SourceReader

  private queue: T[] = [];
  private waiting: ((value: IteratorResult<T>) => void)[] = [];
  private done = false;
  private finalResultPromise: Promise<R>;
  private resolveFinalResult!: (result: R) => void;

  constructor(
    private isComplete: (event: T) => boolean,
    private extractResult: (event: T) => R,
  ) {
    this.finalResultPromise = new Promise((resolve) => {
      this.resolveFinalResult = resolve;
    });
  }

  push(event: T): void {
    if (this.done) return;

    if (this.isComplete(event)) {
      this.done = true;
      this.resolveFinalResult(this.extractResult(event));
    }

    // Deliver to waiting consumer or queue it
    const waiter = this.waiting.shift();
    if (waiter) {
      waiter({ value: event, done: false });
    } else {
      this.queue.push(event);
    }
  }

  end(result?: R): void {
    this.done = true;
    if (result !== undefined) {
      this.resolveFinalResult(result);
    }
    // Notify all waiting consumers that we're done
    while (this.waiting.length > 0) {
      const waiter = this.waiting.shift()!;
      waiter({ value: undefined as any, done: true });
    }
  }

  async *[Symbol.asyncIterator](): AsyncIterator<T> {
    while (true) {
      if (this.queue.length > 0) {
        yield this.queue.shift()!;
      } else if (this.done) {
        return;
      } else {
        const result = await new Promise<IteratorResult<T>>(
          (resolve) => this.waiting.push(resolve)
        );
        if (result.done) return;
        yield result.value;
      }
    }
  }

  result(): Promise<R> {
    return this.finalResultPromise;
  }
}`}

### Queue + Waiter 推拉机制

这个设计的核心在于 `push` 和 `[Symbol.asyncIterator]` 的对称协作：

```
场景 A：生产者先到（push 快于消费）
  push(event)
    → waiting 为空
    → event 入 queue
  next() 被调用
    → queue 有数据
    → 直接 yield queue.shift()

场景 B：消费者先到（消费快于 push）
  next() 被调用
    → queue 为空且未 done
    → 创建 Promise，resolve 存入 waiting
    → 消费者挂起等待
  push(event) 被调用
    → waiting 非空
    → 取出 resolve 直接交付 event
    → 消费者恢复执行
```

**关键洞察**：`queue` 和 `waiting` 永远不会同时非空。要么有待消费的事件（queue 非空），要么有等待事件的消费者（waiting 非空），这是推拉对偶的自然结果。

### 双消费模式

`EventStream` 支持两种互不冲突的消费方式：

```typescript
// 方式 1：for-await-of 逐事件消费（实时 UI 更新）
const stream = streamFunction(model, context, options);
for await (const event of stream) {
  switch (event.type) {
    case "text_delta":
      process.stdout.write(event.delta); // 实时输出
      break;
    case "done":
      console.log("完成:", event.reason);
      break;
  }
}

// 方式 2：await result() 直接等待最终结果（跳过中间过程）
const stream2 = streamFunction(model, context, options);
const finalMessage = await stream2.result(); // 阻塞直到 done/error
console.log(finalMessage.content);
```

**方式 1** 适合 UI 场景：逐 delta 渲染打字机效果。**方式 2** 适合后台场景：不关心中间过程，只要最终结果。两种方式可以组合使用 —— 先用 for-await 迭代完所有事件，再用 `result()` 获取最终消息。

### 事件分组与语义

这 13 种事件可以分为四组：

| 分组 | 事件 | 语义 |
|------|------|------|
| **流控制** | `start` | 流开始，初始化 partial message |
| **文本块** | `text_start` → `text_delta`(n) → `text_end` | 一个文本内容块的完整生命周期 |
| **思维块** | `thinking_start` → `thinking_delta`(n) → `thinking_end` | 一个思维链内容块的完整生命周期 |
| **工具调用块** | `toolcall_start` → `toolcall_delta`(n) → `toolcall_end` | 一个工具调用的参数增量构建 |
| **终止** | `done` / `error` | 流结束，携带完整消息或错误 |

### contentIndex：内容块的定位器

每个非 start/done/error 事件都携带 `contentIndex`，它指向 `AssistantMessage.content[]` 数组中的位置。这是因为一次 LLM 响应可能包含多个内容块：

```
content[0]: thinking  (思维链)
content[1]: text      (回复文本)
content[2]: toolCall  (工具调用 1)
content[3]: toolCall  (工具调用 2)
```

`contentIndex` 让消费者知道当前事件属于哪个内容块，无需维护额外的状态机。

### Interleaved Thinking（交错思维）

在某些模型（如 Claude）的 extended thinking 模式下，思维块和文本块可以交错出现：

```
thinking_start  (contentIndex=0)  ← 第一段思考
thinking_delta  "Let me analyze..."
thinking_end
text_start      (contentIndex=1)  ← 开始回答
text_delta      "Based on my analysis..."
text_end
thinking_start  (contentIndex=2)  ← 第二段思考（交错!）
thinking_delta  "Wait, I should also consider..."
thinking_end
text_start      (contentIndex=3)  ← 补充回答
text_delta      "Additionally..."
text_end
```

`contentIndex` 的递增确保了即使事件交错，消费者仍能正确区分不同的内容块。

### partial：累积状态快照

每个中间事件都携带 `partial: AssistantMessage` —— 这是 **到目前为止已收到的所有内容的完整快照**。它不是 delta，而是全量。这个设计让 UI 渲染非常简单：

```typescript
for await (const event of stream) {
  // 不需要自己维护累积状态
  // event.partial 就是当前完整的 message 快照
  renderMessage(event.partial);
}
```

代价是每个事件都携带了完整的 partial 对象，增加了内存开销。但在客户端这通常不是问题，而 Proxy 场景下可以通过剥离 partial 来优化带宽（详见 4.5 节）。

### Signatures：内容校验

`text_end` 和 `thinking_end` 事件的 `content` 字段包含完整文本，同时 `AssistantMessage` 中的内容块支持 `textSignature` / `thinkingSignature` 字段。这些签名用于 Anthropic 的内容验证机制，确保返回的文本未被中间层篡改。

### StopReason 完整语义

| StopReason | 含义 | 后续动作 |
|------------|------|----------|
| `"stop"` | 模型自然结束 | 返回结果 |
| `"length"` | 达到 max_tokens 限制 | 可能内容被截断 |
| `"toolUse"` | 模型请求工具调用 | Agent 执行工具后继续循环 |
| `"aborted"` | 用户或系统取消 | 清理资源，可用部分内容 |
| `"error"` | 运行时错误 | 重试或报告错误 |

### ToolResult 同步 emit

工具执行结果通过同步 emit 发出，**没有 update 事件**（工具的进度更新通过 `tool_execution_update` 独立报告）：

```typescript
// tool_execution_start: 工具开始执行
await emit({ type: "tool_execution_start", toolCallId, toolName, args });

// 工具执行中...（onUpdate 回调触发 tool_execution_update）

// tool_execution_end: 工具执行完毕，结果立即可用
await emit({ type: "tool_execution_end", toolCallId, toolName, result, isError });

// 紧接着发出 ToolResultMessage
await emit({ type: "message_start", message: toolResultMessage });
await emit({ type: "message_end", message: toolResultMessage });
```

注意 `tool_execution_end` 和 `message_start/end` 是分开的 —— 前者是工具层面的事件，后者是消息层面的事件。这种分层让 UI 可以分别渲染工具执行状态和消息列表。

### abort 后消息可继续使用

`EventStream` 的一个重要设计是：即使 `error` 事件的 `reason` 为 `"aborted"`，返回的 `AssistantMessage` 仍然包含到中止点为止的所有内容：

```typescript
// error 事件中的 message 包含已收到的部分内容
stream.push({
  type: "error",
  reason: "aborted",
  error: partial,  // partial 包含中止前收到的所有 text/thinking/toolCall
});
```

这意味着用户按下 Ctrl+C 后，UI 仍然可以展示已经接收到的回复内容，而不是一片空白。

## 4.4 流式 JSON 解析

### 问题：工具调用参数是不完整的 JSON

当 LLM 流式返回工具调用时，参数以 JSON 片段逐步到达：

```
toolcall_delta: '{"q'
toolcall_delta: 'uer'
toolcall_delta: 'y":'
toolcall_delta: ' "weath'
toolcall_delta: 'er in Tokyo"'
toolcall_delta: ', "max_results": 5}'
```

累积起来是 `'{"query": "weather in Tokyo", "max_results": 5}'`，但在中间的任何时刻，累积的字符串都是 **不完整的 JSON**。UI 需要在参数构建过程中就展示正在输入的内容（"Searching for: weather in To..."），所以需要一个能处理不完整 JSON 的解析器。

### parseStreamingJson 实现

### 设计决策：三层 fallback

为什么不直接用 `partial-json`？因为 **大部分时候 JSON 最终会完整**（`toolcall_end` 时参数已完成），标准的 `JSON.parse` 比 `partial-json` 快得多。三层策略确保了：

1. **完整 JSON** → `JSON.parse`（最快路径）
2. **不完整但可恢复** → `partial-json`（如 `{"query": "wea` → `{"query": "wea"}`）
3. **完全无法解析** → 返回 `{}`（安全降级，永不抛异常）

## 4.5 Proxy 传输层

### 为什么需要 Proxy？

直接从浏览器调用 LLM API 有两个根本问题：

1. **安全性** —— API Key 必须暴露在客户端代码中
2. **CORS** —— LLM API 通常不允许浏览器直接跨域请求

Proxy 模式通过一个中间服务器解决这两个问题：

```
┌──────────┐   POST /api/stream   ┌──────────────┐   stream()   ┌───────────┐
│  Browser  │ ─────────────────── │ Proxy Server │ ──────────── │  LLM API  │
│ (Client)  │                     │  (Server)    │              │           │
└──────────┘                     └──────────────┘              └───────────┘
     ↑                                  │
     │    NDJSON (精简事件)              │    SSE (完整事件)
     │    无 partial 字段               │    含 partial 字段
     └──────────────────────────────────┘
```

### NDJSON 行协议

### EOF 边界处理

`buffer.split("\n")` + `lines.pop()` 是一个经典的行协议解析模式。注意以下边界情况：

```
Chunk 1: '{"type":"text_delta","delta":"he'  ← 一行被截断
Chunk 2: 'llo"}\n{"type":"done",...}\n'      ← 跨越两行

处理过程:
  Chunk 1: buffer = '{"type":"text_delta","delta":"he'
           lines = [], buffer = 原值 (无 \n)
  Chunk 2: buffer = '{"type":"text_delta","delta":"hello"}\n{"type":"done",...}\n'
           lines = ['{"type":"text_delta","delta":"hello"}', '{"type":"done",...}']
           buffer = '' (最后 pop 出空字符串)
```

## 4.7 本章小结

### 核心要点

1. **EventStream 是推拉对偶的异步队列** —— `queue` 和 `waiting` 永远互斥非空，实现了生产者和消费者的速率解耦。双消费模式（`for-await` + `result()`）满足了 UI 实时渲染和后台批量处理两种需求

2. **13 种 AssistantMessageEvent 构成统一的事件协议** —— 所有 provider adapter 将原始 API 事件转换为这套协议，上层代码无需感知不同 LLM API 的差异。`contentIndex` + `partial` 让消费者无需维护状态机

3. **12 种 AgentEvent 形成三层嵌套生命周期** —— Agent → Turn → Message/ToolExecution。`message_update` 是两层事件系统的桥接点，同时携带高层消息和底层事件

4. **流式 JSON 解析采用三层 fallback** —— `JSON.parse` → `partial-json` → `{}`。每次 delta 全量重解析，简单可靠，对于工具参数的典型大小（几十到几百字节）性能足够

5. **Proxy 传输层通过剥离 partial 实现 80% 的带宽节省** —— 客户端用 `processProxyEvent` 从 delta 重建 partial，SSE 行协议 + buffer/split 模式处理 chunk 边界

### 架构关系图

```

  ├──────────────────────────────────────────┐

```

## 从 Demo 到生产：你的 Agent 缺了什么

如果你跟着前面的章节一路走来，你已经在 Demo 06 中亲手构建了一个完整的 coding agent。它能读文件、写代码、执行命令、调用工具。你看着它在终端里流畅地输出代码，心想：这东西真的能用了。

然后你决定让它干一票大的。

> 周五下午，你启动 Agent，给它一个真实任务：把项目里散落在 12 个文件中的旧版 API 调用全部迁移到新版 SDK。Agent 开始工作，你靠在椅背上，看着它逐个文件分析、修改，一切顺利。
>
> **第 15 分钟**，你注意到 Agent 的输出变得奇怪。它开始重复分析已经改过的文件，好像忘了自己五分钟前刚处理过它们。你翻了一下日志——Context Window 满了。Agent 的"记忆"被截断，它丢失了前面一半的工作上下文，只能凭剩下的片段盲目推测。
>
> **第 20 分钟**，Agent 尝试编辑 `config.ts`，报错失败。你排查了五分钟才发现原因：文件里有一对 Unicode 智能引号（`""`），Agent 在 diff 里用了普通引号（`""`），字符串匹配失败。一个肉眼几乎看不出的字符差异，让编辑操作彻底卡死。
>
> **第 25 分钟**，你的猫跳上键盘，终端窗口关闭了。
>
> 所有进度归零。
>
> 你重新打开终端，重新启动 Agent。它礼貌地跟你打了个招呼，像是第一次见面。它不记得任何事——不记得已经改了 7 个文件，不记得哪些 API 已经迁移完毕，不记得你们之间关于边界情况的讨论。你只能从头来过。
>
> 第二周，你把这个 Agent 分享给同事小张。小张用的是不同的 Model、不同的 IDE、不同的项目规范。Agent 在小张手里判若两人——它不遵守小张团队的代码风格，不理解小张项目的目录结构约定，生成的代码在 CI 里全线飘红。
>
> 你的 Agent 没有变笨。它的内核一直在正确工作——它依然能推理、能生成代码、能理解指令。
>
> **它缺的不是智能，是智能之外的一切。**

谁来管理 Context Window 的生命周期？谁来处理 Unicode 字符的边界情况？谁来在会话中断时保存和恢复进度？谁来把项目规范注入 Agent 的工作流程？谁来确保不同用户、不同环境下 Agent 行为的一致性？

这些问题没有一个跟"让 LLM 更聪明"有关。它们属于另一个领域——一个在 Agent 热潮中被系统性忽视的工程学科。

这就是 Harness Engineering 要解决的问题。

## 什么是 Agent Harness

### 多源定义

"Harness"这个词在 Agent 语境中出现的频率越来越高，但不同的团队赋予了它不同的含义。我们先来看看业界怎么说：

| 来源 | 定义 |
|------|------|
| **LangChain** (2026) | "Agent = Model + Harness。Harness 是模型之外的一切——代码、配置、执行逻辑。" |
| **Anthropic** (2025.11) | "使 Agent 能跨多个上下文窗口有效工作的基础设施。" |
| **vtrivedy / HaaS** (2025.09) | "增强模型运行时执行的外部功能集——对话管理、工具调用、权限、会话状态、循环控制、可观测性。" |
| **Martin Fowler** (2026) | "让 AI Agent 保持受控的工具和实践。" |
| **Pi** (README) | "Pi is a minimal terminal coding harness." |

这些定义各有侧重，但指向同一个核心直觉。综合以上来源，本教程给出如下定义：

**Agent Harness 是包裹 Agent 内核的工程系统。内核负责推理，Harness 负责其余一切——让推理变得可控、可靠、可持久、可适应、可交付。**

打一个比方：Model 是引擎，Harness 是汽车。引擎提供动力，但你不能骑着引擎上路——你需要底盘、转向系统、制动系统、仪表盘、座椅。没有人会因为引擎一样就说两辆车一样。事实上，汽车品牌之间的差异化恰恰不在引擎（大家用的都差不多），而在车身工程——底盘调教、安全系统、人机交互、驾驶体验。

Agent 产品也是如此。当所有人都接入同一个 Claude 或 GPT 模型时，产品之间的差距来自哪里？不是 Model，是 Harness。谁的 Context 管理更精准，谁的 Tool 调用更可靠，谁的会话恢复更无缝，谁的用户体验更顺滑——这些决定了产品的成败。

### 为什么叫 Harness

"Harness"这个词的选择不是偶然的，它精确地捕捉了这层工程的本质。

这个词最古老的含义是**马具**——缰绳、鞍具、挽具。马有巨大的力量，但未经驯服的力量是危险的、不可预测的。马具不提供力量，它让力量变得可控、可导向、可安全使用。

在软件工程中，**测试线束**（test harness）是另一个经典用法。测试线束包裹被测代码，为它提供可控的运行环境——模拟输入、捕获输出、管理状态。被测代码本身的逻辑不变，但线束让它的行为变得可观测、可复现。

Agent Harness 继承了这两层含义：它包裹 Agent 内核，使原始的推理能力在确定性的工程世界中安全落地。内核是非确定性的——同样的输入可能产生不同的输出。Harness 的职责是在这种不确定性之上建立可靠性。

三种用法的共同本质是一样的：**Harness 不是力量的来源，而是让力量变得可用的工程结构。**

## 为什么 Harness Engineering 是一门独立学科

有些人觉得 Harness Engineering 不过是"给 Agent 加更多功能"。证据讲的是另一个故事——Harness 对最终结果的影响，远超模型选择本身。三个独立的证据来源证明了这一点。

### 量化证据

> "We improved our coding agent from Top 30 to Top 5 on Terminal Bench 2.0 by only changing the harness."
> —— LangChain, *The Anatomy of an Agent Harness* (2026)

模型没换，Prompt 没换。**25 个排名位次**的跃升，纯粹来自 Harness Engineering。

> 15 个专业工具：80% 准确率，274.8 秒，~102k tokens
> 2 个通用工具（bash + SQL）：**100% 准确率**，77.4 秒，~61k tokens
> —— Vercel (2026)，同一个模型（Claude Opus 4.5）

一个更简单的 Harness 全面优于更复杂的 Harness——不仅准确率更高，速度也更快，token 消耗也更少。

> "These failures were predominantly not knowledge failures. The failures were execution and orchestration problems."
> —— APEX-Agents 基准测试 (2026.01)，前沿模型在真实专业任务上 ~24% pass@1

前沿模型在传统 benchmark 上轻松拿到 **90%+**。从 benchmark 到真实任务的巨大落差，几乎完全是 Harness Engineering 的缺口。模型不缺知识，缺的是可靠的执行与编排基础设施。

> Manus 通过 KV-cache 优化（纯 Harness 层基础设施改进），将缓存 token 成本从 $3/MTok 降到 $0.30/MTok。模型没换。Meta 以 ~$20 亿收购了 Manus。
> —— 收购的不是模型，是 Harness。

### 收敛证据

OpenAI Codex、Claude Code、Manus——三个团队，独立开发，没有互相参考——却收敛到了**相同的 Harness 模式**：

- **更少、更通用的工具**（而非更多专业工具）
- **文件系统作为外部状态**（git + 进度文件）
- **上下文压缩层级**（compaction hierarchies）
- **可扩展的分层架构**（MCP / Extensions / Skills）

这意味着什么？三个独立的收敛证明了一件事：**这个问题空间存在深层结构**。Harness Engineering 有可发现的规律——解决方案不是任意的、拍脑袋的。就像三个独立的实验室在不知道彼此存在的情况下发现了同一个物理定律，这些模式的反复出现证明它们反映的是问题本身的内在约束，而非偶然的工程选择。这正是它能成为一门独立学科的根基。

### 代码比例

Pi 的代码比例直观地说明了这一点。agent-core（内核）约 **~1,900 行**，coding-agent（完整产品）约 **~40,000 行**。比例约 **1:21**。换句话说，Harness 占了产品代码的 **~95%**。内核是冰山一角，Harness 是水面下的庞然大物。

## 理论框架

证据表明 Harness Engineering 很重要。但 Harness **为什么**存在？是什么**力量**催生了它们？如果三个独立团队收敛到了相同的模式，背后一定存在某种深层结构。本节通过四个理论透镜来拆解这个结构。

### 五个根本性鸿沟

每个 Harness 子系统的存在，都是因为 LLM 的本质与现实世界的要求之间存在一道根本性的鸿沟。

| 鸿沟 | LLM 的本质 | 现实世界的要求 | Harness 的桥梁 | 核心 trade-off |
|------|-----------|-------------|--------------|---------------|
| **状态鸿沟** | 无状态函数调用 | 工作是有状态的 | 会话管理、持久化 | 更多状态 = 更多上下文成本 = 更少推理空间 |
| **边界鸿沟** | 在 token 空间思考 | 世界在文件/进程/网络空间运行 | 工具（翻译层） | 更多工具 = 更多能力但更多选择歧义 |
| **信任鸿沟** | 必须给予权力才有用 | 不能完全被信任 | 权限、沙箱、Hook | 更多信任 = 更快但更危险 |
| **注意力鸿沟** | 有限上下文窗口 | 任务产生无界信息 | Compaction、子 Agent、外部记忆 | 更多压缩 = 更多推理空间但更多信息丢失 |
| **适应鸿沟** | 同一个模型权重 | 不同用户/环境/任务 | 配置、Skills、动态 Prompt | 更多灵活性 = 更多配置复杂度 |

每个鸿沟创造了一个 trade-off 空间。Harness Engineering 的本质就是在这五个 trade-off 空间中做出设计决策。三家独立收敛不是巧合——它们都在同一组 trade-off 中寻找最优解，而问题空间的结构约束了解的形状。

这五个鸿沟将在 Part 2 中逐一展开。每个鸿沟直接映射到一个或多个后续章节：状态鸿沟 + 注意力鸿沟 → Ch12（Context 工程），边界鸿沟 + 信任鸿沟 → Ch11（工具与安全），适应鸿沟 → Ch13（配置与 Skills）。而控制循环（Ch10）和交付层（Ch14）分别扮演**协调层**和**呈现层**的角色，将五个鸿沟的解决方案编织成一个完整的产品。

### 成本反转原理

在所有理论透镜中，这一条对设计决策的指导最为直接。它用一个原理解释了大量看似无关的 Harness 设计选择。理解了这个原理，你就能预测一个优秀的 Harness 在面对任何新的子系统设计时"应该"怎么做。

**"在 LLM 上下文中，保留是昂贵的，取回是廉价的——与传统计算机内存模型完全相反。"**

这个反转贯穿了 Harness 的每一个子系统：

| 子系统 | "保留"（昂贵） | "按需取回"（廉价） |
|--------|------------|----------------|
| 工具 Schema | 所有工具定义常驻上下文 | ToolSearch 按需加载 |
| Skills | 所有 Skill 完整内容注入 prompt | 元数据常驻，内容按需加载（渐进式披露） |
| 历史消息 | 全量保留在上下文中 | Compaction 摘要 + 文件按需重读 |
| 文件内容 | 预加载所有相关文件 | read 工具按需读取 |
| 子任务上下文 | 全部在主 Agent 上下文中 | 子 Agent 隔离执行，只返回摘要 |

**在 Harness 的每一层，默认选择"按需取回"而非"常驻保留"，除非有证据表明取回延迟不可接受。**

这个原理将在 Part 2 中反复出现——它解释了为什么 Vercel 的 2 个工具击败了 15 个（Ch11），为什么 Compaction 在 75% 而非 95% 时触发（Ch12），为什么 Skills 使用渐进式披露（Ch13），以及为什么 CLAUDE.md 应该控制在 60 行以内（Ch13）。
